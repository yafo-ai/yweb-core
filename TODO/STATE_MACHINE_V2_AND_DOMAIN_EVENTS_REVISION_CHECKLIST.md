# 状态机 V2 与轻量领域事件设计稿修订清单

本文档用于整理以下两份设计文档当前存在的问题点，并将其收敛为一份可执行的修订清单：

- `TODO/STATE_MACHINE_V2_DESIGN.md`
- `TODO/LIGHTWEIGHT_DOMAIN_EVENTS_AND_OWNSONE_DESIGN.md`

目标不是重新设计方案，而是帮助当前方案完成“收口”，使其在以下方面更加稳健：

- 语义一致
- 默认行为安全
- API 收敛明确
- 与现有 YWeb 能力兼容
- 后续实现和测试更容易落地

---

## 1. 修订目标

本轮修订应优先解决以下问题：

1. 避免状态迁移、状态历史、领域事件三者在“持久化时机”上产生语义错位
2. 避免两份文档对同一能力给出不同说法
3. 避免第一阶段 API 过于灵活，导致实现复杂度失控
4. 明确状态机动作、事务、领域事件之间的职责边界

---

## 2. 修订优先级

| 优先级 | 说明 |
|------|------|
| P0 | 必须先修订，否则方案默认语义存在较大误导风险 |
| P1 | 强烈建议修订，否则实现阶段容易反复返工 |
| P2 | 可后续补充，主要是文档完整性与可读性优化 |

## 3. P0：必须先修订

## P0-1 统一“状态迁移成功”的定义

### 问题

当前文档中，“动作成功”“状态切换成功”“状态历史记录”“领域事件登记”“保存”之间的顺序仍有潜在歧义。

特别是以下组合会让读者误以为：

- 状态还未持久化成功
- 但历史已经记录
- 事件已经登记
- 甚至在无事务场景下可能已触发后续处理

这会导致“业务上看起来成功，但数据库并不一定已确认”的语义风险。

### 修订动作

在两份文档中统一补充一个明确原则：

> 对状态机而言，只有当状态变更已经进入可提交事务语义，才允许把该状态变化视为“可记录历史、可登记领域事件”的成功迁移。

建议明确写死以下规则：

- 关键状态迁移默认要求运行在事务中
- `record_history=True` 与 `event=...` 仅在事务语义下推荐使用
- 若 `save=False` 且当前不在事务中，默认不应把该动作描述为“完整成功迁移”

### 验收标准

文档中必须明确回答以下问题：

- 什么才叫“迁移成功”
- 历史是在什么条件下记录
- 领域事件是在什么条件下登记和发布
- `save=False` 时哪些行为仍然允许，哪些行为不推荐

---

## P0-2 收紧无事务场景下的领域事件默认行为

### 问题

当前轻量领域事件文档中，无事务场景仍允许立即分发，虽然有警告，但默认值仍偏宽松。

这与状态机动作这种高业务敏感场景叠加后，会扩大语义不稳的问题。

### 修订动作

建议在设计文档中把“无事务时立即分发”从默认推荐路径降级为兼容兜底路径：

- 第一阶段仍可保留兼容能力
- 但文档中的主叙述必须改为“事务内使用是标准路径”
- 对状态机动作场景应明确写：推荐或要求在事务中执行

建议文档中加入以下表述：

| 场景 | 建议 |
|------|------|
| 普通模型手工记录事件 | 可兼容无事务立即分发 |
| 状态机动作触发事件 | 推荐强制事务内执行 |
| 涉及历史记录和事件同时发生 | 应在事务内执行 |

### 验收标准

- 两份文档都不再把“无事务立即分发”描述成常规推荐路径
- 状态机章节明确写出：关键迁移动作应放入事务边界

---

## P0-3 去掉第一阶段 `event="OrderPaid"` 这种字符串事件写法

### 问题

状态机文档中仍存在 `event="OrderPaid"` 这类写法。

字符串写法看起来简单，但会引入一整套隐含复杂度：

- 名称解析
- 事件注册表
- 导入时机
- 冲突处理
- IDE 跳转和类型提示缺失

这与当前“轻量领域事件”的方向不一致。

### 修订动作

第一阶段推荐 API 收敛为仅支持两种：

1. 事件类
2. 事件工厂函数

例如：

```python
@transition(
    source=Status.PAID,
    target=Status.SHIPPED,
    event=OrderShipped,
)
```

或：

```python
@transition(
    source=Status.PAID,
    target=Status.SHIPPED,
    event=lambda self, **ctx: OrderShipped(
        order_id=self.id,
        tracking_no=ctx["tracking_no"],
    ),
)
```

字符串事件写法如果仍保留，只能放到“未来可选扩展”中，不能出现在第一阶段推荐示例里。

### 验收标准

- 第一阶段示例中不再出现 `event="OrderPaid"`
- 参数说明中把字符串从“支持项”改为“暂不推荐/未来扩展”

---

## P0-4 明确 `@transition` 方法本身就是完整执行入口

### 问题

当前文档同时支持：

- 直接调用动作方法
- 通过 `execute_action("pay")` 统一入口调用

但没有明确说明二者是否必然走同一条执行管线。

如果不写死，后续实现时非常容易出现：

- `order.pay()` 只执行方法体
- `order.execute_action("pay")` 才走状态机流程

这会让装饰器方法沦为隐形后门。

### 修订动作

在状态机文档中明确加入一条硬规则：

> 被 `@transition` 装饰的方法本身就是完整的状态机执行入口；`execute_action()` 只是基于动作名的分发包装，不允许拥有不同的核心语义。

需要明确两者共享：

- 状态规则检查
- 业务守卫检查
- 权限检查
- 历史记录
- 领域事件登记
- 异常语义

### 验收标准

- 文档中出现显式表述，禁止两套执行语义并存
- 示例中说明 `execute_action()` 最终调用对应 transition action

## 4. P1：强烈建议修订

## P1-1 统一“校验分层”的数量和定义

### 问题

两份文档对校验分层的描述不一致：

- 一份写 4 层：状态、业务守卫、权限、并发
- 一份写 3 层：状态、业务守卫、权限

这会直接影响后续实现、异常分类和测试设计。

### 修订动作

建议统一为 4 层：

1. 状态规则检查
2. 业务守卫检查
3. 权限检查
4. 并发与版本检查

并在两份文档中保持完全一致的顺序和术语。

### 验收标准

- 两份文档对校验分层的数量、名称、顺序完全一致
- 并发检查不再被当作“实现细节”省略掉

---

## P1-2 收敛领域事件推荐 API

### 问题

轻量领域事件文档已经提出了更方便的事件登记方向，但推荐用法还没有完全统一。

当前同时出现以下思路：

- `add_domain_event(Event(...))`
- `event=EventClass`
- `event=lambda ...`

如果没有一个主路径，后续业务代码很快会写散。

### 修订动作

建议统一为以下分层：

- 底层原语：`add_domain_event(event)`
- 业务默认用法：`publish_event(EventClass)` 或等价 API
- 复杂事件：`publish_event(factory_or_event, **kwargs)`

并在状态机文档中与之对应：

- `event=EventClass`
- `event=factory`

### 验收标准

- 文档明确区分“底层原语”和“业务推荐用法”
- 状态机文档与领域事件文档中的示例风格一致

---

## P1-3 明确历史记录与领域事件的相对顺序

### 问题

当前文档虽然提到了“历史入队”“事件入队”，但没有清晰说明二者的稳定顺序和失败策略。

### 修订动作

建议在状态机文档中加入固定表述：

- 历史记录属于事务内持久化数据
- 领域事件属于事务内待发布事实
- 二者都依附于同一事务边界
- 若事务回滚，二者都应视为未发生

同时建议明确：

- 历史记录失败是否阻止迁移
- 领域事件登记失败是否阻止迁移

推荐第一阶段：

- 历史记录写入失败：阻止迁移完成
- 领域事件构造失败：阻止迁移完成
- 领域事件处理器执行失败：不影响已提交事务

### 验收标准

- 文档中给出失败策略表
- “登记失败”和“处理失败”被明确区分

---

## P1-4 收紧 `save=False` 的文档描述

### 问题

`save=False` 是为兼容现有 ORM 使用习惯保留的，但放到状态机 v2 里会被误解成“推荐模式”。

### 修订动作

建议把 `save=False` 定位改成：

- 兼容能力
- 高级用法
- 需要调用方明确理解事务和持久化语义

不应再作为状态机动作的主流示例默认出现。

### 验收标准

- 状态机主示例优先使用事务化执行语义
- `save=False` 被明确标注为兼容/高级选项

---

## P1-5 收敛状态历史字段设计

### 问题

文档中同时出现了 `changed_by` 和 `operator_id`，语义高度重叠。

如果不尽早统一，后续会影响：

- 历史模型字段设计
- API 输出
- DTO 命名
- 审计口径

### 修订动作

建议二选一：

- 要么统一用 `changed_by`
- 要么统一升级为 `operator_id`

如果保留两个字段，必须明确它们的边界：

- `operator_id`：谁触发了动作
- `changed_by`：谁最终导致状态变化落库

但对当前 YWeb 来说，这通常没有必要。

### 验收标准

- 文档中只保留一个主字段名
- 示例和表结构建议保持一致

## 5. P2：建议补充完善

## P2-1 明确异常分类建议

建议文档补充一张“动作失败类型表”：

| 失败类型 | 示例 | 推荐异常语义 |
|------|------|------|
| 非法源状态 | `paid -> pending` 不允许 | 状态规则异常 |
| 守卫失败 | 缺少物流单号 | 业务守卫异常 |
| 权限不足 | 当前用户无 `order.ship` | 权限异常或业务异常 |
| 乐观锁冲突 | 并发重复发货 | 并发冲突异常 |
| 历史记录失败 | 历史模型写入异常 | 迁移失败 |
| 事件处理器失败 | 通知发送失败 | 提交后处理失败，不回滚事务 |

---

## P2-2 明确状态机元数据接口的上下文语义

建议文档明确区分以下两类接口：

- 纯结构接口：只返回动作定义，不考虑当前用户和上下文
- 上下文接口：结合当前对象状态、权限、守卫结果返回可执行动作

例如：

- `Model.get_state_machine_schema()`：纯结构
- `obj.get_available_actions(context=...)`：上下文相关

---

## P2-3 增加“最小可落地版本”定义

建议在状态机文档中补一节“第一阶段最小实现范围”，明确只做：

- `@transition`
- `execute_action()`
- 动作元数据输出
- 与领域事件的基本打通
- 与状态历史的基本打通

明确暂不做：

- 表单 DSL
- 多事件列表
- 分层状态机
- 并行状态机
- 动态流程引擎

这样可以避免实现阶段范围继续膨胀。

## 6. 推荐统一口径

为避免两份文档持续漂移，建议统一采用以下口径：

### 领域事件

- 事务钩子是技术机制
- 领域事件是业务事实
- 领域事件默认推荐在事务内登记、提交后发布

### 状态机动作

- 动作是业务语义入口
- 状态只是动作结果
- `@transition` 方法本身就是完整执行入口

### 历史记录

- 历史属于事务内数据
- 不是提交后副作用
- 应与主状态变更保持一致事务语义

### `save=False`

- 属于兼容能力
- 不是第一阶段推荐路径
- 用于高级调用方自行控制持久化时机

## 7. 推荐修订顺序

建议按以下顺序修订两份文档：

1. 先统一“迁移成功 / 历史 / 事件 / 保存”的时机定义
2. 再统一领域事件 API 和 `event` 参数支持范围
3. 再统一校验分层、异常语义、双入口执行语义
4. 最后补最小落地范围、元数据接口和字段命名细节

---

## 8. 修订完成后的通过标准

如果修订完成，应该满足以下判断标准：

1. 读者能明确知道状态机动作什么时候算真正成功
2. 读者不会误以为无事务 + `save=False` + 事件/历史 是推荐组合
3. 两份文档对领域事件、状态机、事务的关系说法一致
4. 第一阶段 API 足够收敛，不会把实现拖进过度设计
5. 后续测试可以直接围绕文档中的稳定语义编写

---

## 9. 一句话总结

当前方案方向是对的，问题主要不在“思路错误”，而在“默认语义还不够收紧、文档口径还不够统一、第一阶段 API 还略显发散”。

本次修订的核心任务，不是增加更多能力，而是把现有方案压缩成一套：

- 安全的默认语义
- 一致的术语体系
- 可直接实现的最小 API
