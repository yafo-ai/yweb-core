# API First 数据权限扩展说明（Python 版）

## 1. 文档定位

本文是 `API_FIRST_PERMISSION_DESIGN_PYTHON.md` 的**补充说明文档**，不替代原文档，而是在原有“API First 功能权限”方案基础上，进一步扩展出一套可落地的**数据权限**设计。

原文档主要解决的是：

- 用户是否可以调用某个 API
- 前端按钮是否应该显示
- 权限如何以后端 API 为唯一来源

本文解决的是：

- 用户虽然有这个 API 权限，但**能看到哪些数据**
- 用户虽然能执行这个动作，但**能对哪些记录执行**
- 同一个资源在不同动作下，**数据范围如何不同**

一句话区分：

- **功能权限**回答的是：`能不能做`
- **数据权限**回答的是：`能对哪些数据做`

## 2. 为什么要把数据权限单独抽出来

如果只做功能权限，会遇到一个非常典型的问题：

- 用户拥有 `user:list`
- 但他不应该看到所有用户

或者：

- 用户拥有 `order:approve`
- 但他只能审批自己区域的订单

或者：

- 用户拥有 `customer:update`
- 但他只能修改自己负责的客户

这说明：

- 功能权限只决定“门开不开”
- 数据权限决定“门打开后能走到哪里”

所以数据权限不应该被塞进按钮权限里，也不应该简单地等同于角色权限，而应该作为一层独立能力补进原来的 API First 模型中。

## 3. 本文聚焦的场景范围

本文只聚焦你最常见、也最值得优先抽象的三类维度：

### 3.1 组织维度

- 本人
- 本部门
- 本部门及下级
- 指定部门集合
- 全部

### 3.2 关系维度

- 创建人
- 负责人
- 跟进人
- 项目成员
- 审批链参与者
- 客户归属人

### 3.3 动作维度

同一资源在不同动作上数据范围不同，例如：

- 可以 `list` 本部门用户
- 但只能 `update` 自己创建的用户
- 可以 `read` 所有订单
- 但只能 `approve` 自己区域订单

这三类维度，已经覆盖了绝大多数中后台、CRM、项目管理、工单系统的核心数据权限问题。

## 4. 扩展原有 API First 方案的核心原则

原文档里，API 权限的核心是：

- 后端 API 是唯一权限源
- 权限声明只写在后端
- OpenAPI 输出权限元数据
- 前端只消费元数据，不手写权限码

扩展到数据权限时，仍然坚持同样的原则：

### 4.1 数据权限也以后端为唯一来源

前端不参与数据权限判断逻辑，不拼接过滤条件，不做真实数据裁剪。  
前端最多只做一些体验层提示，例如：

- “你只能查看本部门数据”
- “你只能编辑自己创建的记录”

真正的数据范围必须以后端判定为准。

### 4.2 数据权限附着在 API 动作上，而不是页面上

不要把数据权限定义成：

- “用户管理页可看本部门”
- “订单详情页可看全部”

而应该定义成：

- `user:list` 的数据范围是本部门
- `user:update` 的数据范围是本人创建
- `order:approve` 的数据范围是负责区域

因为页面是 UI 概念，动作才是业务概念。

### 4.3 数据权限是“动作上下文中的过滤规则”

不要把数据权限理解成一个孤立的标签。  
它更准确的表达应该是：

> 针对某个用户，在调用某个 API 动作时，系统应该附加什么数据过滤规则。

所以数据权限的核心不是一个名字，而是一个**可计算、可组合、可执行的规则集合**。

### 4.4 功能权限先判断，数据权限后收敛

执行顺序应始终是：

1. 先判断用户是否有 API 调用权
2. 再判断这次调用允许接触哪些数据

也就是：

- 没有 `order:approve`，直接拒绝
- 有 `order:approve`，再判断只可审批哪些订单

## 5. 数据权限在整体架构中的位置

扩展后，原来的架构可以升级为：

```text
请求进入 API
  -> 功能权限校验：是否允许调用当前 API
  -> 数据权限解析：得到当前动作下的数据范围
  -> 查询过滤 / 单条校验 / 批量校验
  -> 返回结果
```

也就是说，数据权限不是取代功能权限，而是插在功能权限之后的一层：

- `Permission Guard`：判断能不能做
- `Data Scope Resolver`：判断能做哪些数据

## 6. 数据权限的抽象模型

为了兼容组织、关系、动作三类维度，建议把数据权限抽象成三层模型：

### 6.1 动作层

表示正在做什么，例如：

- `list`
- `read`
- `create`
- `update`
- `delete`
- `approve`
- `assign`
- `export`

### 6.2 范围层

表示数据范围来源于什么维度，例如：

- 组织范围
- 关系范围
- 全量范围
- 自定义范围

### 6.3 规则层

表示最终可执行的过滤规则，例如：

- `owner_user_id = current_user.id`
- `dept_id = current_user.dept_id`
- `dept_id in current_user.subtree_dept_ids`
- `creator_id = current_user.id`
- `leader_id = current_user.id`
- `region_id in current_user.region_ids`
- `customer_owner_id = current_user.id`

换句话说：

- 动作层决定“在哪个 API 语境中”
- 范围层决定“从哪个维度取范围”
- 规则层决定“最终如何过滤数据”

## 7. 组织维度设计

这是最常见、最稳定的一类数据权限。

## 7.1 组织维度的标准档位

建议统一成以下五档：

- `self`
- `dept`
- `dept_and_children`
- `dept_set`
- `all`

对应语义：

| 标识 | 含义 |
| --- | --- |
| `self` | 仅本人 |
| `dept` | 本部门 |
| `dept_and_children` | 本部门及下级部门 |
| `dept_set` | 指定部门集合 |
| `all` | 全部 |

## 7.2 组织维度不等于用户表所属部门

这里要注意一点：  
组织维度最终过滤的字段，不一定永远是 `dept_id`。

不同资源可能组织挂靠点不同，例如：

- 用户：`dept_id`
- 订单：`biz_dept_id`
- 客户：`owner_dept_id`
- 项目：`belong_dept_id`

所以组织维度要拆成两个概念：

1. **组织范围档位**
2. **当前资源的组织锚点字段**

例如：

- 对 `user:list` 来说，组织字段是 `dept_id`
- 对 `order:approve` 来说，组织字段可能是 `region_dept_id`

不能把“组织权限”写死成只过滤某个固定字段。

## 7.3 组织维度适用的动作

最适合组织维度的动作通常是：

- `list`
- `read`
- `export`

部分情况下也适用于：

- `update`
- `delete`
- `approve`

但在修改、审批等动作上，组织维度往往需要和关系维度一起组合，而不是单独使用。

## 8. 关系维度设计

关系维度不是按组织树，而是按业务关联来判断。

这类权限在 CRM、项目管理、工单系统里尤其常见，而且很多时候比组织维度更重要。

## 8.1 关系维度的标准关系类型

建议先统一支持以下几种：

- `creator`
- `owner`
- `follower`
- `member`
- `approver`
- `customer_owner`

对应语义：

| 标识 | 含义 |
| --- | --- |
| `creator` | 创建人 |
| `owner` | 负责人 |
| `follower` | 跟进人 |
| `member` | 项目成员 |
| `approver` | 审批链参与者 |
| `customer_owner` | 客户归属人 |

## 8.2 关系维度的难点

关系维度比组织维度复杂，原因在于：

- 同一关系在不同资源上的字段不同
- 有些关系是一对一
- 有些关系是一对多
- 有些关系不在主表上，而在关联表上

例如：

- 创建人可能是 `created_by`
- 负责人可能是 `owner_id`
- 项目成员可能在 `project_member` 关联表
- 审批链参与者可能在 `workflow_task` 表

所以关系维度不能只靠“字段名约定”，还要允许“关系解析器”。

## 8.3 关系维度的推荐建模方式

建议把关系维度抽象成：

- **关系类型**
- **关系解析方式**

关系解析方式至少支持两种：

1. **字段直连**
   - 例如 `created_by = current_user.id`
   - 例如 `owner_id = current_user.id`

2. **关联关系**
   - 例如当前用户是否在项目成员表里
   - 例如当前用户是否出现在审批任务表里

也就是说，关系维度不一定总能翻译成“主表上的一个字段比较”，它可能是一段更复杂的业务关系判断。

## 8.4 关系维度适用的动作

关系维度最适合：

- `read`
- `update`
- `delete`
- `approve`
- `assign`

尤其在这些动作上，关系维度往往比组织维度更自然、更符合业务语义。

例如：

- “我能查看我负责的客户”
- “我能修改我创建的记录”
- “我能审批流转到我这里的单据”
- “我是项目成员，所以我能查看项目任务”

## 9. 动作维度设计

动作维度是本文最重要的一层，因为它决定了：

> 同一个资源，不同动作是否允许使用不同的数据范围。

答案必须是：**允许，而且应该允许。**

## 9.1 为什么动作维度必须独立

如果不把动作维度抽出来，你会得到很粗糙的权限定义，例如：

- “用户数据范围 = 本部门”

但真实业务通常不是这样。真实业务更像：

- `user:list = dept`
- `user:read = dept_and_children`
- `user:update = creator`
- `user:delete = self`

或者：

- `order:list = dept`
- `order:read = all`
- `order:approve = region_owner`

所以正确的抽象不是：

- 资源级别的数据权限

而是：

- **资源 + 动作级别的数据权限**

## 9.2 动作维度的推荐表达

建议把动作数据范围定义为：

```text
<resource>:<action> -> <data scope policy>
```

例如：

- `user:list -> dept`
- `user:update -> creator`
- `customer:read -> owner or follower`
- `order:approve -> dept_set`

也就是说，数据权限的直接挂点不是“资源”，而是“动作”。

## 9.3 动作维度的价值

动作维度一旦引入，你就能自然支持这些常见规则：

- 可以看本部门，但不能改本部门全部数据
- 可以看所有订单，但不能审批所有订单
- 可以读项目，但只能操作自己参与的项目
- 可以导出部门内数据，但不能导出全量

这正是很多系统里最重要、也最容易被忽略的一层。

## 10. 推荐的数据权限策略模型

基于上面的三类维度，建议你把数据权限最终建模为：

### 10.1 Policy

一个动作对应一个或多个数据策略。

例如：

- `user:list` 绑定一个策略
- `customer:read` 绑定多个策略
- `order:approve` 绑定一个复杂策略

### 10.2 Strategy

一个策略描述：

- 使用哪个维度
- 采用什么规则
- 规则之间如何组合

### 10.3 Predicate

最终可执行的条件表达式。

简单理解可以是：

- 一个策略是“业务规则”
- 最后会被翻译成“过滤条件”

## 10.4 推荐的最小结构

每条策略至少包含：

| 字段 | 说明 |
| --- | --- |
| `resource` | 资源名 |
| `action` | 动作名 |
| `scope_type` | 范围类型，如 `org` / `relation` / `all` |
| `scope_value` | 范围值，如 `dept` / `creator` |
| `effect` | 通常为 `allow` |
| `priority` | 优先级 |
| `operator` | 多条规则之间的组合方式，如 `or` / `and` |

例如概念上可以表达为：

- `user:list -> org:dept`
- `user:update -> relation:creator`
- `customer:read -> relation:owner OR relation:follower`
- `project:read -> relation:owner OR relation:member`

## 11. 数据权限的几种可扩展实现方式

基于原文档的 API First 设计，数据权限可以扩展出几种实现路线。

## 11.1 方式一：动作绑定单一数据范围

最简单，最适合先落地。

例如：

- `user:list -> dept`
- `user:update -> creator`
- `order:approve -> dept_set`

优点：

- 清晰
- 简单
- 易实现

缺点：

- 复杂场景不够灵活

适合第一阶段。

## 11.2 方式二：动作绑定复合规则

例如：

- `customer:read -> owner OR follower`
- `project:read -> owner OR member`
- `task:update -> creator OR owner`

优点：

- 能覆盖大部分 CRM / 项目场景

缺点：

- 规则组合复杂度上升

适合第二阶段。

## 11.3 方式三：动作绑定上下文策略解析器

例如：

- `order:approve` 由审批流模块自己解析“我是否是当前审批节点参与者”
- `project:read` 由项目模块解析“我是否在项目成员表中”

优点：

- 最灵活
- 最贴近业务

缺点：

- 通用性略弱
- 对基础设施要求更高

适合复杂关系维度场景。

## 11.4 推荐路线

建议分层推进：

1. 先做**组织维度 + 单一动作范围**
2. 再补**关系维度 + 复合规则**
3. 最后对少量复杂动作引入**专用解析器**

这样扩展性和工程复杂度最平衡。

## 12. 如何与原有 OpenAPI 元数据模型衔接

原文档里，API 主要输出：

- `x-permission`
- `x-resource`
- `x-action`

扩展到数据权限后，建议增加一组**说明性元数据**，用于前端理解和调试，但**不把前端当成真正执行方**。

建议新增：

- `x-data-scope-enabled`
- `x-data-scope-mode`
- `x-data-scope-summary`

示意含义如下：

| 字段 | 含义 |
| --- | --- |
| `x-data-scope-enabled` | 当前 API 是否启用数据权限 |
| `x-data-scope-mode` | 当前 API 主要使用哪类范围，如 `org` / `relation` / `mixed` |
| `x-data-scope-summary` | 人类可读的摘要，例如“本部门及下级”或“负责人或跟进人” |

注意：

- 这些字段主要用于说明、调试、管理台展示
- 不建议前端依据这些字段自己执行真正过滤

真正的过滤逻辑仍然后端执行。

## 13. 如何与 RBAC 模型衔接

原文档里，功能权限推荐的是：

- `users`
- `roles`
- `permissions`
- `user_roles`
- `role_permissions`

扩展到数据权限后，建议增加一层：

- `role_data_policies`

也可以理解成：

- 角色拥有功能权限
- 角色还拥有该功能下的数据范围策略

即：

- `role -> permission`
- `role + permission -> data policy`

这样就能支持：

- 角色 A 拥有 `user:list`，且数据范围是 `dept`
- 角色 B 也拥有 `user:list`，但数据范围是 `all`

这比把数据范围直接塞到角色表一个字段里更合理，因为：

- 同一角色对不同动作的数据范围可能不同

## 14. 推荐的数据库方向

本文不展开具体表结构实现细节，但建议数据模型至少支持以下能力：

### 14.1 角色与功能权限关联

已有即可。

### 14.2 角色与动作级数据策略关联

必须支持：

- 角色
- 资源
- 动作
- 范围类型
- 范围值

### 14.3 指定部门集合

因为你明确提到“指定部门集合”是高频场景，所以建议专门支持：

- 某条数据策略绑定多个部门

而不是临时把它塞进 JSON 文本里。

### 14.4 复杂关系策略的扩展槽

因为“项目成员”“审批链参与者”这类场景很难纯靠固定字段表达，所以建议预留：

- 业务关系类型
- 关系解析器标识
- 模块自定义参数

这能避免未来重新推翻模型。

## 15. 组织维度、关系维度、动作维度如何组合

这是设计成败的关键点。

## 15.1 推荐组合方式

建议你把组合规则设计成：

- **动作优先**
- 动作下再看采用哪个范围维度
- 同一动作可支持多个范围规则组合

也就是：

1. 先确定当前是哪个动作
2. 再确定这个动作的数据范围策略
3. 再把策略翻译成最终过滤条件

## 15.2 常见组合示例

### 示例一：用户管理

- `user:list -> org:dept`
- `user:read -> org:dept_and_children`
- `user:update -> relation:creator`

含义：

- 用户列表只能看本部门
- 用户详情可以看本部门及下级
- 用户修改只能修改自己创建的记录

### 示例二：CRM 客户

- `customer:list -> relation:owner OR relation:follower`
- `customer:read -> relation:owner OR relation:follower`
- `customer:update -> relation:owner`
- `customer:assign -> org:dept_set`

含义：

- 客户列表和详情可看自己负责或跟进的客户
- 客户修改只能由归属人执行
- 客户分配只允许处理指定部门范围内的客户

### 示例三：项目管理

- `project:list -> org:dept_and_children OR relation:member`
- `project:read -> relation:owner OR relation:member`
- `project:update -> relation:owner`

含义：

- 项目列表允许看到本部门及下级项目，也允许看到自己作为成员参与的项目
- 项目详情允许负责人和成员查看
- 项目修改仅负责人可操作

### 示例四：审批单据

- `order:list -> org:dept`
- `order:read -> all`
- `order:approve -> relation:approver`

含义：

- 列表只看本部门
- 详情允许更大范围读取
- 审批只能由当前审批链参与者操作

这正是动作维度重要性的最佳例子。

## 16. 数据权限在不同 API 类型上的执行方式

不同 API 类型，数据权限执行点不同。

## 16.1 列表接口

例如：

- `GET /users`
- `GET /orders`

这是数据权限最主要的场景。  
执行方式一般是：

- 自动附加查询过滤条件

组织维度和关系维度都很适合在这里应用。

## 16.2 详情接口

例如：

- `GET /users/{id}`
- `GET /orders/{id}`

详情接口不能只靠“先查出来再返回”，而应该：

1. 先判断是否有 `read` 权限
2. 再判断这条记录是否在当前用户的数据范围内
3. 不在范围内则返回无权限或不可见

## 16.3 更新接口

例如：

- `PUT /users/{id}`
- `PATCH /orders/{id}`

这类接口非常适合动作维度差异化控制。

常见规则：

- 可看很多，但只能改很少
- 可查全部详情，但只能修改自己负责的数据

## 16.4 删除接口

删除通常应当比查看更严格。  
推荐删除动作单独配置数据范围，不要直接沿用 `read` 或 `list`。

## 16.5 审批接口

审批通常不能简单套组织权限，更多依赖关系维度：

- 是否当前审批节点参与者
- 是否在审批链中
- 是否负责该区域/该业务线

所以审批动作建议优先支持关系维度解析。

## 17. 推荐的执行顺序

为了让实现稳定、可解释，建议统一执行顺序如下：

### 17.1 先做认证

- 用户是否登录

### 17.2 再做功能权限

- 是否有 `resource:action`

### 17.3 再解析动作级数据策略

- 当前动作对应哪种数据范围

### 17.4 再执行数据过滤

- 列表接口：加查询条件
- 详情/修改/删除/审批：校验目标记录是否落在允许范围内

### 17.5 最后执行业务逻辑

这样能保证：

- 权限模型分层清晰
- 问题定位容易
- 日志审计可解释

## 18. 面向你关注场景的推荐方案

如果只围绕你最常见的三类维度，我建议直接采用下面这套路线。

## 18.1 第一阶段：先支持动作级组织维度

先实现：

- `self`
- `dept`
- `dept_and_children`
- `dept_set`
- `all`

并且把它挂到：

- `resource + action`

这是最稳、收益最高的一步。

## 18.2 第二阶段：补充关系维度

优先支持：

- `creator`
- `owner`
- `follower`
- `member`
- `approver`
- `customer_owner`

其中建议分两类：

- 简单字段类：`creator`、`owner`
- 复杂关系类：`member`、`approver`

复杂关系类建议允许模块级解析器介入。

## 18.3 第三阶段：允许动作使用复合规则

例如：

- `customer:read -> owner OR follower`
- `project:read -> owner OR member`
- `task:update -> creator OR owner`

这样你就能覆盖绝大多数真实业务。

## 18.4 第四阶段：对审批类动作引入业务关系解析器

审批类场景往往不是固定字段，而是流程上下文。  
这一层建议作为专项扩展，而不是一开始就抽得过泛。

## 19. 你最应该避免的几种做法

为了让这套设计真的能长期扩展，下面几种做法建议避免。

## 19.1 不要把数据权限继续绑回页面按钮

错误方向：

- 页面 A 的编辑按钮可改本部门数据
- 页面 B 的编辑按钮可改本人数据

这样会重新回到 UI 驱动的权限模型，失去 API First 的价值。

正确方向：

- `resource:action` 决定数据范围

## 19.2 不要把所有数据权限塞成角色表一个字段

例如：

- 角色只有一个 `data_scope = dept`

这会导致：

- `list`、`read`、`update`、`approve` 全部共用同一个范围

而这恰恰不符合你的高频场景。

正确方向：

- 数据范围要挂在“动作级权限”上

## 19.3 不要强行把关系维度都映射成部门

例如：

- 客户归属人
- 项目成员
- 审批链参与者

这些都不是组织树问题。  
如果强行按部门处理，会造成大量业务偏差。

## 19.4 不要让前端负责真正的数据裁剪

前端只能做提示，不能做真实安全控制。  
真正的数据权限过滤必须在后端执行。

## 20. 与原文档的关系总结

你可以把两份文档合起来理解：

### 原文档负责

- API 功能权限
- OpenAPI 权限元数据
- 前端 API 自动感知权限
- 按钮基于 API 自动显示/禁用

### 本文负责

- API 数据权限
- 组织、关系、动作三维扩展
- 动作级数据范围策略
- 列表/详情/修改/审批等动作的数据过滤原则

两者叠加后，形成完整闭环：

1. 是否允许调用 API
2. 调用了之后可以作用于哪些数据

## 21. 最终结论

基于你最关心的场景，最推荐的数据权限扩展方向是：

- **功能权限继续走 API First**
- **数据权限也挂在 API 动作上**
- **组织维度、关系维度只是数据范围来源**
- **动作维度决定最终使用哪种数据范围**

其中最关键的设计结论只有一句：

**数据权限不要定义成“某角色的数据范围”，而要定义成“某角色在某资源的某动作下的数据范围”。**

如果只记住这一句，后面整个模型就不容易跑偏。
